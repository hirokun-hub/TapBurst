## 1. タップカウント — デバイス公平性のための時間ベース制御

### 合意事項（信頼度: 非常に高い — 3名全員合意）

**CADisplayLinkのフレームレートはデバイスにより異なる（60Hz / 120Hz ProMotion）ため、スコア加算はフレームレート非依存の時間ベースで制御すべき。**

- `preferredFrameRateRange` で60fps固定するアプローチは非推奨（ProMotionの滑らかな描画を犠牲にする）
- 推奨: スコア加算に最小間隔 `1/60秒 = 16.67ms` を設定し、描画は端末のネイティブレートに任せる
- これにより理論上限は常に **600タップ/10秒** で固定される

### 実装方式

専門家間で2つの方式が提案された:

| 方式 | 提案元 | 概要 | 利点 |
|------|--------|------|------|
| pendingCount + tick消費 | O | touchesBeganでカウント加算、displayLinkで1だけ消費 | デバッグしやすい |
| タイムスタンプ比較 | A, G | touchesBegan内で `now - lastValidTapTime >= 16.67ms` なら即+1 | フレーム落ちに強い、CADisplayLinkに依存しない |

**推奨: タイムスタンプ比較方式**（2名が推奨、フレーム落ち耐性が高い）

### 参考資料

- [Apple Developer Documentation: CADisplayLink](https://developer.apple.com/documentation/quartzcore/cadisplaylink)
- [Apple Developer Documentation: preferredFrameRateRange](https://developer.apple.com/documentation/quartzcore/cadisplaylink/preferredframeraterange)

## 2. Finish画面 — タッチ無効化の実装

### 合意事項（信頼度: 非常に高い — 3名全員合意）

**SwiftUIの `allowsHitTesting(false)` だけではUIViewRepresentable経由のタッチイベントを防げない。UIView側でのブロックが必須。**

- `allowsHitTesting(false)` はSwiftUIのヒットテストから除外するが、UIKitのUIViewが直接タッチを受け取るケースをカバーしない
- 入力の最上流（UIViewの `touchesBegan`）でゲームフェーズを確認し、`.finish` 中はイベントを破棄するのが最も確実

### 推奨アプローチ（堅牢順）

1. `touchesBegan` の先頭で `guard gamePhase == .playing else { return }` — 入力源で遮断
2. SwiftUI側で `allowsHitTesting(false)` — 二重防御
3. 必要に応じて透明な吸収レイヤー（`contentShape(Rectangle())` + ジェスチャ）

### Finish画面の表示時間

- 専門家O: 0.8〜1.2秒（推奨1.0秒）
- 専門家A: 1.5〜2.0秒
- 専門家G: 1.5〜2.0秒
- **合意範囲: 1.0〜2.0秒**（タップ運動の慣性は約0.8〜1.2秒で収束するという知見に基づく）

### スコアカウントアップアニメーション

- **推奨: 0.7〜1.0秒、easeOut系イージング**（3名の推奨範囲が重なる領域）
- 線形よりeaseOut（終盤減速）が「気持ちよさ」を感じやすい

## 3. 人間のタップ速度 — 学術的ベンチマーク

### 合意事項（信頼度: 非常に高い — 学術研究に基づく）

**単指タッピングの学術的ベンチマーク:**

| 指標 | 値 | ソース |
|------|-----|--------|
| 右手単指 10秒タッピング | 平均 6.31 tps（インターバル 164ms） | Finger Tapping Test研究 |
| FEPSY神経心理テスト標準値 | 平均 55〜60回/10秒（5.5〜6.0 tps） | FEPSY標準値 |
| エリートeスポーツ選手（単指） | 7.4 Hz（7.4 tps） | ResearchGate |

**スマホゲーム文脈での推定値（複数指交互タップ）:**

- 一般ユーザー: 4〜8 tps（40〜80/10秒）
- ゲーム慣れ: 8〜15 tps（80〜150/10秒）
- 熟練者: 15〜25 tps（150〜250/10秒）
- 超人域: 25〜40 tps（250〜400/10秒）

> 注意: 「30〜40 tps = 人間の実用上限」という初期仮定は両手10本指フル活用の極限値。1フレーム1カウント制約下では複数指の同時タップがスコアに寄与しないため、実際のスコア分布はこれより低くなる可能性が高い。**称号テーブルはプレイテスト後に実データで調整が必須。**

### 参考資料

- Finger Tapping Test（10秒単指タッピング研究） — 平均インターバル・tps/secの具体値
- FEPSY神経心理テスト標準値（健常成人 55〜60回/10秒）
- ResearchGate: エリートeスポーツ選手タップ速度研究（7.4 Hz）

## 4. 画面揺れ — カーブ選定と制御方式

### 合意事項（信頼度: 非常に高い — 3名全員合意）

#### 4-1. 時間カーブ: pow(t, 3)（3乗）を推奨

`t = elapsed / 10.0` として:

| 時点 | pow(t,2) | pow(t,3) | exp(5t)正規化 |
|------|----------|----------|---------------|
| 6秒 | 0.360 | 0.216 | 0.129 |
| 8秒 | 0.640 | 0.512 | 0.364 |
| 9秒 | 0.810 | 0.729 | 0.604 |
| 10秒 | 1.000 | 1.000 | 1.000 |

- 3乗カーブは8秒あたりから急上昇する感覚が自然で、「後半一気感」の演出に最適
- 指数カーブ（k=5）はより急峻だが、8秒まで演出が弱すぎて「盛り上がり」が薄くなるリスク
- 調整が必要ならpow(t, 4)も選択肢

#### 4-2. 合成方式: 乗算

**`amplitude = maxAmplitude * tapRateFactor * timeFactor`**（3名全員合意）

- 乗算は「時間 x 頑張り = 演出強度」という直感に一致
- タップしなければ後半でも揺れない → ゲームとプレイヤーの一体感
- 下駄を履かせる変形（`0.2 + 0.8 * tapFactor`）も検討可（専門家Oの提案）

#### 4-3. 揺れパターン: sin波ベース推奨（ランダムは非推奨）

**毎フレーム完全ランダムな方向は「ノイズ的な細かい振動（ジャダー）」になりやすく、不快感を誘発する。**（3名全員合意）

推奨アプローチ:
- X軸とY軸で異なる周波数のsin波を合成（リサージュ図形的アプローチ）
- 周波数は 1.5〜2.0 Hz 程度が「緊張感」と「快適性」のバランスが良い

### maxAmplitude

- 専門家O: ±2〜4pt
- 専門家A: ±5〜8pt（推奨±8pt）
- 専門家G: ±5pt
- **安全範囲: ±3〜5pt**（iPhone SE 375pt高さでの可読性を考慮）

### タップレート正規化上限

- 専門家O: 15〜20 tps
- 専門家A: 25 tps
- 専門家G: 20 tps
- **推奨範囲: 20〜25 tps で 1.0 にクランプ**

## 5. パーティクル演出

### 合意事項（信頼度: 非常に高い）

#### 5-1. scaleRange

**推奨: base scale の ±30〜60%**（3名の推奨範囲が重なる領域）

| 専門家 | 推奨範囲 |
|--------|----------|
| O | ±30〜60% |
| A | ±60〜80% |
| G | ±40〜50% |

- ±100%（scaleRange = base）は極小〜極大になり「ゴミっぽさ」が出やすい（専門家O）
- まずは ±40〜50% から始めて調整

#### 5-2. scaleSpeed の限界（信頼度: 非常に高い — Apple API仕様）

**`scaleSpeed` は線形（一定速度）の変化のみ。単一CAEmitterCellで「膨らんで→縮む」は不可能。**（3名全員合意）

代替手法:
- 2種類のセルに分離（短命膨張セル + 長命縮小セル）
- 実用的な代替: 大きめのscaleで生成 → `scaleSpeed` マイナス + `alphaSpeed` マイナスで「縮みながら消える」

#### 5-3. CAEmitterCell.color は動的変更可能（信頼度: 非常に高い — Apple API仕様）

- Animatableプロパティとして公式ドキュメントに記載
- ただし実行中の動的変更はCore Animationキャッシュとの不整合リスクあり
- **推奨: Tier変更時はセルを作り直して `emitterCells` を差し替える**（専門家A, O）

#### 5-4. パフォーマンス目安

同時パーティクル数 = `birthRate x lifetime x 同時エミッター数`

| 専門家 | 安全域（iPhone SE基準） |
|--------|----------------------|
| O | 同時1000〜1500個以下 |
| A | 同時150個以下（非常に保守的） |
| G | 同時1000〜1500個以下 |

- 現在の最大値: 64 x 0.5 x 5 = 160個（十分余裕あり）
- **推奨: 同時1000個以下を目安に設計**（O, Gが合意。Aの150個は過度に保守的）
- velocity引き上げ時はlifetimeとbirthRateのバランスに注意

- [Apple Developer Documentation: CAEmitterCell](https://developer.apple.com/documentation/quartzcore/caemittercell)
- [Apple Developer Documentation: CAEmitterLayer](https://developer.apple.com/documentation/quartzcore/caemitterlayer)

## 6. パーティクルとスコアの視覚的不整合

### 合意事項（信頼度: 高い — 2名合意、1名は別見解）

5本指タッチで5箇所にパーティクルが出るがスコアは+1、という状況について:

- **専門家O, A**: 問題なし。パーティクルは「触った手触り（フィードバック）」、スコアは「ルール上のカウント」としてプレイヤーは分けて理解する
- **専門家G**: 誤ったフィードバックになりうる。有効タップにのみパーティクルを出すべき

> 判断: O, Aの見解を採用。パーティクルは全タッチ位置に出す（触覚的フィードバックとして自然）。ただし、プレイテストで混乱が見られた場合はGの方式も検討。
