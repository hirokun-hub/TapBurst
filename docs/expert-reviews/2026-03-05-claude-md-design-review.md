## 1. CLAUDE.md の読み込み仕様（全専門家一致、信頼度97%以上）

### 4階層のメモリシステム

| レイヤー | パス | スコープ |
|----------|------|----------|
| Enterprise policy | システムディレクトリ | 組織全体 |
| Project memory | `./CLAUDE.md` or `./.claude/CLAUDE.md` | チーム共有 |
| User memory | `~/.claude/CLAUDE.md` | 個人の全プロジェクト |
| Project local | `./CLAUDE.local.md` | 個人のこのプロジェクトのみ（gitに入れない） |

- 全メモリファイルはセッション起動時に自動ロードされる
- サブディレクトリの CLAUDE.md は、Claude がそのサブツリーのファイルを読んだときにオンデマンドロードされる
  - ただし、実装やバージョン差でサブディレクトリが読まれない報告もある（専門家O指摘）。重要ルールはルート or @import で確実に入れるべき

ソース: https://docs.anthropic.com/en/docs/claude-code/memory

### @import 機構

- CLAUDE.md 内で `@path/to/file` 構文により他ファイルをインポート可能
- 相対パス・絶対パスどちらも使用可能
- 最大5段階の再帰インポートに対応
- インポートされたファイルは起動時に展開されてコンテキストに読み込まれる

### 行数制限に関する重要な区別

- **CLAUDE.md 自体に公式の200行制限は存在しない**（専門家A指摘）。200行制限は Auto Memory の `MEMORY.md`（`~/.claude/projects/<project>/memory/MEMORY.md`）にのみ適用される仕様
- ただし CLAUDE.md が長いほど遵守率が低下するのは全専門家が一致して指摘
- LLM が信頼できる形で遵守できる指示数は約150〜200程度。Claude Code のシステムプロンプト自体が約50指示を消費するため、実質的に CLAUDE.md で使える枠はさらに限られる（専門家A指摘）
- 実用推奨: 50〜120行程度（全専門家の推奨を総合）

## 2. グローバル vs リポジトリの責務分離基準（全専門家一致）

| 指示の性質 | 配置先 |
|------------|--------|
| 全プロジェクト・全言語で常に正しい | グローバル (`~/.claude/CLAUDE.md`) |
| 特定スタック（Swift/iOS等）にのみ適用 | リポジトリ (`./CLAUDE.md`) |
| 個人の都合（秘密・端末依存） | `./CLAUDE.local.md` |

### 「定数の局所化」ルールについて
- 全専門家が「リポジトリ側が適切」と回答
- Swift/iOS 固有の設計判断であり、非Swift プロジェクトではノイズになる

### 折衷案（専門家O提案）
- `~/.claude/swift-ios.md` を作成し、Swift専用ルールを集約
- TapBurst の `./CLAUDE.md` から `@~/.claude/swift-ios.md` でインポート
- 非Swift プロジェクトのコンテキストを汚さない

## 3. 設計書が重厚な場合の CLAUDE.md 構成パターン（全専門家一致）

### 原則: CLAUDE.md は「索引 + 絶対ルール」に徹する

- 設計書の内容を転記しない（二重管理回避）
- 技術的制約は1行チェックリスト化して記載し、詳細は設計書参照
- 要件定義の正本（source of truth）を必ず明記する

### ドキュメントバージョン競合の防止
- 正本を明示し、古いバージョンを「参照しない」と明記する
- `IMPORTANT:` 等の強調を使うことが推奨される（専門家G指摘）

## 4. 実装未着手プロジェクトの初期 CLAUDE.md（全専門家ほぼ一致）

### 初期に書くべきもの
1. プロジェクト概要（1〜2行）
2. 技術スタック・ターゲット
3. 破ると設計が崩れる絶対制約（1行チェックリスト形式）
4. ドキュメントの正本指定とパス
5. 開発者のレベル（iOS未経験であること）
6. ビルド・テストコマンド

### 後から追加すべきもの
- ディレクトリ構成（ソースが増えてから）
- Claude が繰り返し間違えるパターン（「〜はするな」形式）
- デバッグで判明した注意点

### 定期的な剪定
- 不要になった初期記述は容赦なく削除する（専門家G指摘）
- CLAUDE.md はコードと同様にリファクタリングするもの

## 5. 分割戦略

### 推奨構成

```
./CLAUDE.md                          <- メイン（50〜80行目標）
  @.kiro/specs/tapburst-game/design.md  <- 設計書（@importで必要時参照）
  @~/.claude/swift-ios.md               <- Swift共通ルール（オプション）
```

### 注意点
- @import は起動時に展開されてコンテキストを消費する
- 64KB の設計書を @import すると毎回全文ロードされるため、トークン効率が悪い
- 設計書は @import ではなく「パスを記載して必要時に Claude に読ませる」方が効率的（専門家A指摘）
